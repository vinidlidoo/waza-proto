## Completed

- [17 — Encoded pass-through freeze recovery (PLI deadlock)](completed/17-encoded-freeze-recovery.md) — root-caused the multi-second "frozen until refresh" freezes as **PLI deadlock** and fixed the bytestream (Stage 1: parameter sets injected only at true IRAPs → worst-freeze 3,068 → 411 ms, PLI = 0, DAT GOP measured at 2.93 s). Established the permanent constraint **DAT 0.7.0 exposes zero encoder control** (no keyframe/GOP/IDR/bitrate API; native `PassthroughVideoEncoder` only relays the PLI to an app with no encoder — so native ingest wouldn't fix freezes either). A/B then showed the **in-app re-encode (now H.265 @ 4 Mbps cap) is the better live experience** — snappier, ~half the jb latency, no GOP catch-up jumps, relay-free — at the cost of modest second-generation (tandem-coding) softness. **Final: Path A (re-encode) is the default; Path B (pass-through) kept flag-gated for a possible future return.** Quality bar = "no *visible* tax", not "zero transcode."
- [15 — Encoded-frame ingest (HEVC pass-through)](completed/15-encoded-frame-ingest.md) — dropped the glasses-path HEVC decode + LiveKit H.264 re-encode in favour of Path B (livekit-cli TCP relay, shipped Jan 2026) over Path A (rust-sdks#1048 native API, ~3–6+ months + Swift port). Built end-to-end across three stages (viewer HEVC compat → iPhone TCP listener → Go relay + A/B measurement) and **retained flag-gated** (`Config.glassesEncodedIngest = false`). Stage 2's freeze regression carried into [plan 17](completed/17-encoded-freeze-recovery.md), where it was root-caused (PLI deadlock) and the final ship decision landed on the re-encode default.
- [16 — Encoded-frame smoother (publisher-side jitter buffer) — ABANDONED](completed/16-encoded-smoother.md) — proposed PTS-paced ring buffer on iPhone before TCP send. Three implementation iterations (single-slot, drop-newest, drop-oldest-non-IDR with bitstream IDR detection) all made the viewer worse than the unsmoothed baseline. Architectural finding: smoothing is the wrong layer — (1) lk-cli already paces, so adding iPhone-side smoothing double-paces and any drop we introduce is one lk-cli would have absorbed; (2) the real cause of the 9× worst-freeze regression in encoded mode is **PLI deadlock** — lk-cli has no way to generate keyframes on demand when the viewer's decoder requests them, so freezes only clear on browser refresh. Re-encode wins on freezes specifically because the encoder responds to PLI. **Code deleted in the plan-17 ship** (the finding is the artifact); `containsIRAP` survives, repurposed by plan 17.

- [14 — Viewer fixed aspect box](completed/14-viewer-fixed-aspect-box.md) — locked the browser `<video>` to a fixed 9:16 slot at 504×896 native (CSS-only) so DAT resolution swaps stop reflowing the page. No pipeline change, no added latency. Sharpness/grain follow-up tracked separately as a `features.md` entry.
- [13 — Glasses "disconnected" status message](completed/13-glasses-disconnected-status.md) — replaced the misleading "Grant Camera Access" gate row (which fired whenever the glasses were off-link) with a "Don glasses to connect" status label on the Glasses tab; "Disconnected" still shown on the Front camera tab. Also swapped `VideoView.layoutMode` `.fit → .fill` so both feeds fill the preview container at the same size without black bars.
- [12 — Glasses smoothing buffer](completed/12-glasses-smoothing-buffer.md) — ring buffer + `CADisplayLink` pump @ 30 fps between DAT-decoded frames and `BufferCapturer.capture(...)`, masking BT Classic burstiness identified in plan 11. Apples-to-apples sweep at `{0, 2, 4}` (depth=6 skipped on latency-budget grounds) picked depth=2 as winner: **78% viewer-freeze reduction (45 → 10), 0.0% unique-frame loss at the encoder, ~100 ms latency added**. Depth=0 retained as Config knob escape hatch for when WDAT upstream improves. Full write-up: [docs/glasses-stream-buffer-sweep.md](docs/glasses-stream-buffer-sweep.md).
- [11 — Video quality profiling](completed/11-video-quality-profiling.md) — staged profiler that started with LiveKit sender/receiver/playout stats, then added glasses DAT/decode counters. Stage 2 + Meta-docs verification + `.medium` config sweep proved BT Classic delivery cadence (median 23.8 fps, bursts to 49 fps, stalls to 695 ms) is the root cause of glasses stutter; lowering the requested resolution made things worse (encoder drops doubled). Fix promoted to [plan 12](completed/12-glasses-smoothing-buffer.md).
- [10 — Publisher JWT minted at connect time](completed/10-jwt-auto-refresh.md) — security/ergonomics infra (not a build-ladder rung). New Vercel `/api/publisher-token` endpoint mints LiveKit JWTs on demand, gated by HS256 envelopes signed in the iOS app via `PublisherTokenClient`. `Secrets.swift` no longer carries a baked-in 6h JWT; `refresh-secrets.sh` writes only signing seeds. Stage 3 (in-band rotation loop) skipped after research found LiveKit server already pushes `refresh_token` signals to connected clients and the Swift SDK has no public `updateToken()` — the "6h TTL caps session length" premise was wrong; only initial connect is gated. Full finding in plan body's Decisions logged.
- [09 — App icon](completed/09-app-icon.md) — shipped a real WazaProto iOS app icon using the `path-mark.png` direction. Added light/dark/tinted 1024px PNGs to the asset catalog, verified Xcode generated app icon outputs, and confirmed the icon on the iPhone 17 simulator home screen. Closes v0.09.
- [08 — Automated test suite (local)](completed/08-test-suite.md) — cross-cutting infra (not a build-ladder rung). Five staged tiers, all locally-runnable: Vercel token-mint (Vitest), iOS pure-logic XCTest, iOS XCTest against Meta's MockDeviceKit, iOS XCUITest via MDK's test server, browser viewer Playwright + `lk` publisher. Stages 3 and 4 re-scoped to smoke-only after empirical finding that `stream.videoFramePublisher` doesn't fire under MDK on iOS 26.5 sim — upstream issue [#197](https://github.com/facebook/meta-wearables-dat-ios/issues/197). CI integration tracked separately in [features/ci-integration.md](features/ci-integration.md).
- [07 — Background streaming (glasses path)](completed/07-background-streaming.md) — iPhone keeps publishing the glasses POV while the app is backgrounded or the screen is locked. Required four load-bearing knobs in concert: `UIBackgroundModes` (`audio` + Meta's BLE/MFi modes), `RoomOptions(suspendLocalVideoTracksInBackground: false)`, a published mic track to activate `AVAudioSession`, and a codec swap to `VideoCodec.hvc1` + in-app `VTDecompressionSession` with runtime decoder-rebuild on DAT's adaptive-ladder resolution swaps. Closes v0.07.
- [06 — Shareable viewer link](completed/06-shareable-viewer.md) — viewer hosted at `waza-proto.vercel.app`; Vercel Node serverless token mint gated by per-invite HS256 JWTs (3h TTL, no denylist); iPhone "Copy viewer link" button + "N watching" overlay badge. Closes v0.06.

Closes v0.05.
