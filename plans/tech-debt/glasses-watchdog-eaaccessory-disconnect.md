# Glasses stream: watchdog misses link-level disconnects

> Filename keeps the old `eaaccessory` slug for link stability, but the mechanism
> below was corrected (2026-05-29): `EAAccessory` is the wrong API — see "Where to start."

**What:** A silent Bluetooth link drop of the Ray-Ban glasses (out of range, powered off) doesn't always promote to DAT's `DeviceSession.stateStream` / `errorStream`, so the app's `onTerminated` teardown never fires. The UI keeps showing "Connected" while the glasses are detached. `DeviceSessionState` has no `disconnected` case (only `idle/starting/started/paused/stopping/stopped`), so the watchdog — which consumes only the session state/error streams — has no terminal signal for a link loss that doesn't emit `.stopped`. Path-independent: affects both the shipped re-encode path and the flag-gated pass-through path identically.

**Why deferred:** Pre-existing since plan 13, which wired the *session-level* teardown paths (hinge-fold and session-terminated → `stateStream`/`errorStream` → `onTerminated`) but not a separate availability/link-loss signal. The common case is already covered — Meta's spec confirms *"Closing the hinges disconnects Bluetooth, stops active streams, and forces `SessionState` to `STOPPED`,"* which the watchdog catches. This is the rarer "link drops without an accompanying `.stopped` transition" case (Meta's checklist lists *"`STOPPED` **or loss of availability**"* as the two distinct release triggers). Surfaced again during the plans 15–17 encoded-mode work, but it predates and outlives that decision. Never reproduced — the SDK may emit a `.unexpectedError("Device unavailable")` on some real drops, which the watchdog's error branch already catches, which is likely why it hasn't bitten.

**What would trigger paying it down:** A session where the glasses drop at the link level (out of range, powered off) and the UI stays stuck on "Connected" with a frozen last frame — particularly if it confuses a demo or a real wearer.

**Where to start:** The fix is small (~5–10 LOC) and reuses a stream the app already has. **Do not use `EAAccessoryManager`/`EAAccessoryDidDisconnect`** (an earlier note suggested this) — that's the External Accessory (MFi) framework. The DAT SDK is built on CoreBluetooth over BT Classic, the glasses are not MFi `EAAccessory` devices, and the app has no MFi entitlement, so that observer would never fire. Instead, add a third branch to the `withTaskGroup` in `GlassesSource.startWatchdog` (`GlassesSource.swift:294`) that consumes the injected `deviceSelector.activeDeviceStream()` and returns (→ `onTerminated()`) when it yields `nil` during a live session. `GlassesGateway` (`GlassesGateway.swift:66`) already consumes this same stream and watches `activeDeviceID` go `nil` on a drop, so the signal is proven to fire. The documented alternative is `Wearables.shared.deviceForIdentifier(deviceId).addLinkStateListener { … }` reacting to `LinkState.disconnected`.
